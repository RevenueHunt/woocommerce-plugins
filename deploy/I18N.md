# Translations (i18n) — pipeline & how to add a locale

How the plugin's **own admin/UI strings** are localized. (Shopper-facing quiz
content is localized in the monolith for V1 / the `quiz-html` artifact for V3 —
**out of scope** here.)

## Architecture

| File | Where | Role |
|---|---|---|
| `shared/languages/<canonical-slug>.pot` | committed | Translation **template**. Regenerated from `src/`. Ships in **both** targets. |
| `shared/languages/<canonical-slug>-<locale>.po` | committed | **Source** translations (one per bundled locale), in the canonical eCommerce identity. |
| `…-<locale>.mo` | **build output** (never committed) | Compiled catalog. Generated per target by the build. |

Two distributions, two strategies:

- **eCommerce (wp.org, free)** — on [translate.wordpress.org](https://translate.wordpress.org/projects/wp-plugins/product-recommendation-quiz-for-ecommerce/) (GlotPress). WordPress auto-delivers community language packs since 4.6, so this target ships **only the `.pot`**; we don't bundle `.mo`.
- **WooCommerce (marketplace, free)** — **not** on GlotPress, so it must **bundle `.mo`**. The `woocommerce` target sets `"bundleTranslations": true` in its `target.json`; the build token-substitutes each `.po` so its msgids match the target's transformed source strings, then compiles `.mo` with the pure-PHP `tools/po2mo.php` (no gettext / WP-CLI needed to build).

The `I18n` class loads the text domain on `plugins_loaded`; headers `Text Domain`
/ `Domain Path: /languages` are set in the main file.

## Tooling

| Command | Does | Needs |
|---|---|---|
| `composer i18n:pot` | Regenerate `<slug>.pot` from `src/` (`tools/make-pot.sh`). | WP-CLI (skips gracefully if absent) |
| `composer i18n:lint` | Fail on split / fragmented translatable strings (`tools/i18n-lint.php`). | PHP only |
| `composer i18n:pot-check` | Fail if the committed `.pot` drifted from `src/`. | WP-CLI |
| `composer build` | Builds artifacts; compiles bundled `.mo` for opted-in targets. | PHP only |

**WP-CLI** ([wp-cli.org](https://wp-cli.org/)) is a dev/build prerequisite for
`make-pot` only. CI provisions it; locally, either install `wp` or drop the
phar at `./.wp-cli.phar` (gitignored). `msgfmt`/gettext is **not** required —
`po2mo.php` is the compiler.

CI gates: `i18n:lint`, `i18n:pot-check`, and the loading test
(`tests/Characterization/TranslationLoadingTest.php`).

## When you change or add a string in `src/`

1. Make the change. Keep sentences **whole** — inject links/values with
   `sprintf()` `%s` / `%1$s` placeholders, never by chaining translation calls
   (the `i18n:lint` gate enforces this).
2. `composer i18n:pot` — refresh the template (CI fails otherwise).
3. For each bundled locale, carry existing translations onto the new template:
   `msgmerge --no-fuzzy-matching shared/languages/<slug>-<locale>.po shared/languages/<slug>.pot -o shared/languages/<slug>-<locale>.po`
   then translate the newly-empty `msgstr` entries.
4. `composer build && composer test` — green.

## Add a new locale (e.g. `sv_SE`)

```bash
SLUG=product-recommendation-quiz-for-ecommerce
# a) seed from GlotPress (export the locale's .po) OR start blank from the template:
msginit --no-translator -i shared/languages/$SLUG.pot -l sv_SE \
        -o shared/languages/$SLUG-sv_SE.po
# b) translate the msgstr entries (preserve %s / %1$s placeholders).
# c) add 'sv_SE' to the BUNDLED list in TranslationLoadingTest.php.
# d) build + verify — the .mo is produced automatically (build globs *.po):
composer build && composer test
```

The Woo target bundles **every** `shared/languages/*.po` automatically — no
build edit needed. Validate a `.po` with `msgfmt --check --check-format` (the
`--check-format` pass catches dropped or mismatched `%s` placeholders).

## V3 reuse

This pipeline is generic. The future V3 WooCommerce plugin reuses it verbatim
for its own chrome strings: set `"bundleTranslations": true` in its
`target.json`, drop `.po` files in `shared/languages/`, and the same
`make-pot` → `po2mo` → lint/freshness/loading-test machinery applies. (V3
shopper-facing quiz strings stay in the `quiz-html` artifact — not here.)
